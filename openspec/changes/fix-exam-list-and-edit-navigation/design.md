# Design: Fix Exam List and Edit Navigation

## Technical Approach

Five independent layers: backend redirects (swap 4 URL strings + extract `_redirect_with_flash` into a shared helper used by both `pages.py` and `exams.py`), template cancel links (header `page_actions` + in-form cancels across 5 templates), CSS centering (media-query rule + removal of inline `max-width` on 3 cards), dirty-tracking JS (event delegation on `[data-track-dirty]` + `data-cancel-for` interception + `#unsaved-modal` reusing delete-modal CSS, with a real focus trap added to both modals), and reorder fetch (intercept POST → JSON body → toast → redirect). No new dependencies. No session middleware. Mostly additive or string-replacement, with two deliberate, documented refactors: (a) hoisting `showToast`/`escapeHtml`/`toastContainer` out of the delete-modal IIFE to global scope in `app.js`, and (b) extracting `_redirect_with_flash` into a shared helper module — the earlier "zero structural refactors" claim is retracted for `app.js` and `exams.py`.

## Architecture Decisions

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| 1 | Post-save redirect targets | Exams/questions → list (`/exams`, `/questions`); Answers → `/questions/{qid}`; Image upload → context-aware via hidden `return_to` input (review queue or exam detail) | Entities with standalone lists benefit from list landing; child entities without lists must stay on parent. `return_to` preserves the user's existing upload-flow context without inventing session state |
| 2 | Cancel link principle | Every cancel link uses an explicit hardcoded URL, never `history.back()`. OCR correction → `/questions` replaces unsafe `javascript:history.back()`. BOTH the `page_actions` header link AND the in-form Cancel must be updated in every touched template — four templates have two cancel links: `exams/form.html` (header line 5 `/exams/{{ exam.id }}` AND in-form line 40 `{{ '/exams/' + exam.id|string if exam else '/exams' }}`, which for the edit branch resolves to `/exams/{id}` detail and MUST become `/exams` to satisfy the "Exam edit cancel navigates to exam list" scenario), `manual_form.html` (header line 6 `/exams/{{ exam.id }}` AND in-form line 68), `correct.html` (line 6 AND line 48, both `javascript:history.back()`), and `answers/form.html` (header line 6 AND in-form line 48) | `history.back()` depends on browser history and can land on external sites; explicit URLs are predictable and consistent. The duplicate-link reality must be stated up front or one cancel link will silently bypass the dirty guard (see decision 3) and the `history.back()` (or wrong-target detail URL) will survive in one of the two slots |
| 3 | Dirty tracking + cancel interception | `input`/`change` event delegation on `document`, matching `[data-track-dirty]` forms; set `data-dirty="true"` on first modification. Cancel-link interception is TWO-TIER so neither header nor in-form cancels bypass the guard. Tier 1 (header cancels): every header `page_actions` cancel `<a>` is OUTSIDE its `<form>`, so `link.closest('form')` returns null; each header cancel link gets `data-cancel-for="<form-id>"` and the IIFE intercepts all `[data-cancel-for]` clicks, looks up the form via `document.getElementById(link.dataset.cancelFor)`, and routes to `openUnsavedModal(link.href)` when that form is `data-dirty="true"` (otherwise navigates normally). Tier 2 (in-form cancels): in-form Cancel `<a>` tags live INSIDE the `<form>` and are reachable via `link.closest('form[data-track-dirty]')`; the same IIFE intercepts those clicks, resolves the form via `closest`, and routes to `openUnsavedModal(link.href)` when the form is `data-dirty="true"` (otherwise navigates normally). Affected in-form cancels: `exams/form.html:40`, `questions/manual_form.html:68`, `questions/correct.html:48`, `answers/form.html:48` | No MutationObserver needed (these forms have no dynamically injected fields). Event delegation avoids per-form listeners. The `data-cancel-for` attribute contract is required for Tier 1 because the header cancel lives outside the `<form>` DOM subtree — `closest('form')` cannot reach it; Tier 2 reuses `closest` for the in-form case so a single listener covers both. Affected header links: `exams/form.html:5`, `questions/manual_form.html:6`, `questions/correct.html:6`, `answers/form.html:6`, `questions/bulk_upload.html:6` |
| 4 | Unsaved-changes modal | New `#unsaved-modal` in `base.html` with `.modal`, `aria-modal="true"`, `role="dialog"`, backdrop-click-to-close, Escape key — identical structure to delete modal. Wired in `app.js` IIFE. Adds a real Tab-trap (keydown handler cycling focus within the dialog) AND focus restoration to the trigger on close, in BOTH the delete modal and `#unsaved-modal` | Reuses the delete modal's CSS and markup patterns. NOTE: the existing delete modal only provides focus *management* (initial focus on open via `confirmBtn.focus()`), NOT a focus *trap* — it has no Tab-cycling keydown handler and `closeModal()` never restores focus to the trigger. Because `aria-modal="true"` is present, screen readers announce an inert background that is not actually inert; this a11y gap would be inherited by `#unsaved-modal` unless a real trap is added. The trap + restoration are therefore added to both modals as part of this change |
| 5 | "Guardar y salir" behavior | Triggers native `form.submit()` — modal closes, browser POSTs, server redirects with query-param flash. No async save-then-navigate path | Keeps the existing POST-redirect-flash contract intact; no need for a hybrid JS-toast-after-ajax pattern |
| 6 | Image upload `return_to` | Hidden `<input type="hidden" name="return_to">` in bulk_upload template; `upload_exam_image` reads it from form data via `return_to: str | None = Form(None)` and passes it to `redirect_with_flash` ONLY on the success path. ALL `upload_exam_image` redirect paths (the success path AND all 5 error paths: no-filename validation, OCR-failed, FileValidationError, StorageError, generic Exception — currently bare `RedirectResponse(url=f"/exams/{exam_id}?message=…{str(e)}&type=error")`) are converted to `redirect_with_flash` so the `urllib.parse.quote` mitigation applies uniformly to every flash message, including the non-ASCII `ó` in the no-filename message (`No se proporcionó archivo`, line 145-148) AND the `str(ocr_err)`/`str(e)` strings W2 was filed against. ALL 6 flash-bearing redirect paths are enumerated: success (`:205`) + 5 errors — no-filename validation (`:145`, `No se proporcionó archivo` with non-ASCII `ó`), OCR-failed (`:165`, `str(ocr_err)`), FileValidationError (`:210`, `str(e.message)`), StorageError (`:216`, `str(e.message)`), generic Exception (`:222`, `str(e)`). The bare `NotFoundError` redirect at line 141 (`url="/exams"`, status 302) has NO flash message — it is left as a bare redirect that needs no conversion (no `?message=…` → nothing for `urllib.parse.quote` to encode; converting it would add a flash message that was not there, which is out of scope). `return_to` is consulted ONLY on the success path; error paths always use the safe context-aware default (`/exams/{id}` or `/search/needs-review`), never `return_to`. `return_to` validation: parse the value with `urllib.parse.urlsplit` and REJECT any value whose `.netloc` or `.scheme` is non-empty (kills `//evil.com` and `https://evil.com`); match the **path** component against an allowlist of exact `/search/needs-review` OR the regex `/exams/\d+` via `re.fullmatch` (so only exact `/exams/{digits}` paths pass — `/exams/5/extra` is rejected; path-only match, so a `?exam_id=…` query string does not cause rejection); on any failure, fall back to the context-aware default. The query string of `return_to` is PRESERVED on redirect. The `exam_id` query param, if present in `return_to`, is PINNED server-side from the route's `exam_id` path param (overriding any client-supplied value) to prevent exam-id spoofing. Fallback for missing `return_to`: current context-aware logic | Explicit over implicit. No referrer-based guessing. `return_to` is a client-controlled hidden input routed straight into `RedirectResponse(url=...)` — without validation it is an open-redirect vector (see Risks). The path-only allowlist + netloc/scheme rejection closes it; pinning `exam_id` from the path param prevents cross-exam spoofing; scoping `return_to` to the success path prevents error-path abuse; converting all error paths to `redirect_with_flash` ensures W2's URL-encoding reaches every flash string, not just the success path |
| 7 | Answer reorder fix | JS `fetch` interceptor on `#reorder-form` submit → POST `{"ordered_ids": [int, ...]}` with `Content-Type: application/json` (matching the `ReorderRequest` schema) → `showToast('success', …)` → `window.location` redirect to `/questions/{qid}` (`qid` parsed from the form `action` URL or a `data-question-id` attribute). The hidden `ordered_ids` input is pre-filled from DOM order on page load (the drag-drop JS is the only thing that populates it, so a no-JS load would otherwise submit an empty value). `<form>` keeps `method="POST"` for no-JS fallback | `showToast`/`escapeHtml`/`toastContainer` currently live INSIDE the delete-modal IIFE (`app.js:113-445`, `showToast` at line 358) and are NOT callable from a new IIFE; this design hoists them to global scope (attached to `window`) as a deliberate refactor (see decision 9). HONEST no-JS state: the reorder endpoint takes `payload: ReorderRequest` (Pydantic, expects JSON), but the form posts `application/x-www-form-urlencoded` with `ordered_ids` as a JSON string → FastAPI returns 422. The current no-JS path is ALREADY a broken 422, not a working "raw JSON" path. This design recommends option (a): document that no-JS users cannot reorder and remove the false "degrades to raw JSON" claim; the spec scenario "Reorder degrades without JavaScript" is FLAGGED for the orchestrator as needing amendment. Option (b) — adding a server-side url-encoded `ordered_ids` branch returning a redirect — is rejected as out of scope for this change |
| 8 | CSS centering | `.form-card { max-width: 520px; margin-inline: auto; }` inside `@media (min-width: 1024px)` in `styles.css`. Applied by adding `form-card` to the WRAPPING `<div class="card">` (→ `<div class="card form-card">`) on 3 templates — NOT on the `<form>` element, because the `<form>` lives INSIDE `<div class="card">` and adding `class="card"` to it would create a card-in-card (double border + double background, a visible regression). The `<form>` gets ONLY `data-track-dirty` + `id` (see File Changes). Inline `style="max-width:…"` is REMOVED from the wrapping div on `answers/form.html:10` (560px), `questions/bulk_upload.html:10` (560px), and `exams/form.html:9` (520px) so `.form-card` owns the width | No `.mx-auto` utility exists in the codebase. `margin-inline: auto` is logical-property CSS (better RTL support). Gated by media query so mobile remains full-width. Inline `max-width` wins over the class rule (inline styles beat class rules), so the 560px cards would override `.form-card`'s 520px unless the inline style is removed — removal unifies the width. The `.card` rule (`styles.css:86`) defines `background`, `border`, `border-radius`, `overflow: hidden`; nesting a `.card` `<form>` inside a `.card` div would double those visual properties |
| 9 | Shared JS + Python helpers (deliberate refactors) | JS: hoist `showToast`, `escapeHtml`, and `toastContainer` out of the delete-modal IIFE to module/global scope (attached to `window`) in `app.js` so the new dirty-tracking IIFE can call them. Python: extract `_redirect_with_flash` from `app/api/pages.py` into a shared helper module `app/api/_flash.py` (renamed without the leading underscore, e.g. `redirect_with_flash`), imported by both `pages.py` and `exams.py`; the helper MUST `urllib.parse.quote` both `message` and `msg_type` before concatenation. `Form` is imported in `exams.py` and `return_to: str \| None = Form(None)` is added to the `upload_exam_image` signature | Two files (`app.js`, `exams.py`) are NOT purely additive/string-replacement changes. `showToast` is IIFE-scoped today and cannot be reused as-is; the earlier "reuses existing showToast" + "zero structural refactors" claims are retracted for these files. The flash helper currently concatenates `f"{url}?message={message}&type={msg_type}"` with no encoding; OCR error strings embed `str(ocr_err)`/`str(e)` (may contain `&`, `#`, spaces, non-ASCII) → malformed redirect URLs, so URL-encoding is added as part of the extraction |

## Data Flow

```
┌──────────────────────────────────────────────────────┐
│ BACKEND (pages.py / exams.py)                        │
│                                                      │
│  POST handler ──→ _redirect_with_flash(url, msg)     │
│    exam_create  → /exams                             │
│    exam_edit    → /exams                             │
│    manual_q     → /questions                         │
│    ocr_correct  → /questions                         │
│    image_upload → validated return_to or default     │
│                        │                             │
│  ┌─────────────────────┘                             │
│  │  RedirectResponse 303 ?message=&type=             │
│  ▼                                                    │
│ list.html ──→ _get_flash_from_query ──→ .alert       │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│ FRONTEND (Vanilla JS)                                │
│                                                      │
│  [data-track-dirty] form                             │
│    │ input/change event delegation                   │
│    ▼                                                  │
│  data-dirty="true"                                   │
│    │ Cancel button click                              │
│    ▼                                                  │
│  openUnsavedModal() ──→ #unsaved-modal shown         │
│    │                                                  │
│    ├─ "Guardar y salir" → form.submit() → POST       │
│    ├─ "Descartar"       → window.location = href     │
│    └─ "Cancelar" / Esc  → closeModal()               │
│                                                      │
│  #reorder-form submit (reorder page)                 │
│    │ fetch POST JSON → showToast → redirect          │
│    └─ no-JS: form posts url-encoded → 422 (broken)    │
└──────────────────────────────────────────────────────┘
```

## Migration / Rollout

No migration required. Most changes are additive or string replacements within existing routes. No database schema changes. No new dependencies. Two files (`app.js`, `exams.py`) involve deliberate, documented refactors (helper hoisting + helper extraction) rather than pure string replacement; one new file (`app/api/_flash.py`) is created.

## Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Open redirect via `return_to` | HIGH | `return_to` is a client-controlled hidden input routed into `RedirectResponse(url=...)`. An attacker can craft a form posting `return_to=//evil.com` to redirect the victim off-site after a legitimate upload. Mitigation: parse `return_to` with `urllib.parse.urlsplit`, reject non-empty `.netloc`/`.scheme`, match the path component against the allowlist (`/search/needs-review` exact OR `/exams/\d+` regex via `re.fullmatch`), preserve the query string, and pin `exam_id` from the route path param; see decision 6 for full logic. `return_to` is consulted on the SUCCESS path only; error paths always use the safe default (see decision 6) |
| Malformed redirect URLs from un-encoded flash messages | MEDIUM | OCR error strings embed `str(ocr_err)`/`str(e)` which may contain `&`, `#`, spaces, non-ASCII → broken `?message=…&type=…` query strings. Mitigation: `redirect_with_flash` `urllib.parse.quote`s both `message` and `msg_type` (see decision 9) |
| False a11y announcement from `aria-modal` without a trap | MEDIUM | `aria-modal="true"` makes screen readers announce an inert background; without a Tab trap that inertness is a lie. Mitigation: a real Tab-cycling keydown handler + focus restoration is added to both modals; `aria-modal` is retained only once the trap exists (see decision 4) |
| No-JS reorder is a broken 422, not a graceful degradation | LOW | The current no-JS path 422s today. This design recommends documenting that no-JS users cannot reorder (option a); the spec scenario "Reorder degrades without JavaScript" is FLAGGED for the orchestrator as needing amendment (see decision 7) |

## Open Questions

- **FLAG for orchestrator (W1):** the spec scenario "Reorder degrades without JavaScript" is factually false — the no-JS path 422s today. This design recommends amending the spec to drop the degradation scenario and state that reorder requires JavaScript. Orchestrator must confirm before implementation, since the spec (not just this design) needs to change.
